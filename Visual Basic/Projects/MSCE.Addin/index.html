Add new features

  1. Right-click the project node in Solution Explorer and select Add>New Item.
  2. In the Add New Item dialog box, expand the Bentley node under Visual C# or Visual Basic.
  3. Choose from the available item templates:
    1. Class
    2. Key-In Command Table
    3. Placement Tool
    4. Placement Tool Settings Form
    5. Placement Tool Settings (Wpf)
    6. Scan Criteria Extensions
    7. Selection Tool
    8. Selection Tool Settings Form
    9. Selection Tool Settings (Wpf)
    10. UserControl (Wpf)
    11. Window Form
    and then click Add.
    • If you choose MicroStation Key-In Command Table and a Command.xml and KeyinCommands.cs or KeyinCommands.vb already exist, you will get an error stating "Command table already implemented. Only one command table per project."
    • If a command table exist, you will be prompted to "Add item to keyin". Click Yes to add the item to a first level key-in. If you click No, the item will not be added to the command table. By clicking Yes, you will modify the Command.xml and KeyinCommands.cs or KeyinCommands.vb files. Due to these internal changes you may initially get a

      "Microsoft Visual Studio - Security Warning"
      "The following template is requesting to open one or more documents in a manner that could pose a security risk."

      Because you trust us, click Trust to trust the template and the changes will be made. if you click Do Not Trust, then no change will be made to Commands.xml and the change will have be made manually. Changes to KeyinCommands.cs or KeyinCommands.vb will be made even if you clicked Do Not Trust.

The files for the template that you selected are added to the project. You can start adding functionality to your item template, press F5 to run the project, or add additional item templates.

Run and Debug

To run the project, press F5. Visual Studio will:

  • Build the application addin for the project.
  • By default project settings, the application addin will be saved to the Bentley product mdlapps\ folder. If your project is not set to save the application in the default Bentley mdlapps folder, the MS_MDLAPPS MicroStation configuration variable must be set to the folder in which the application is located.
  • When debugging, start an instance of the Bentley product.

In the instance of the Bentley product you can test out the functionality of your extension/Addin. In MicroStation CONNECT Edition from the main ribbon, select the Utilities tab and in the Utilities group select MDL Applications and click Browse. In the Select MDL Application file dialog box change File of Type: to All Files (*.*). Your addin application will have the extension .dll. Browse to and select your dll addin application and click open. MicroStation will display a message in the MicroStation Message Center. If successfull, you will receive MDLTaskId Loaded and your application will appear in the MDL dialog box Loaded Applications list box. If not successful, you will receive MDLTaskId did not load... and your application will not appear in the MDL dialog box Loaded Applications list box. Select your MDLTaskId. MDLTaskId is the MDL Task Id as defined by your application. If your application has an associated keyin command table, the MDL dialog box Key-ins... button will be enabled. Select the Key-ins... button, if enabled. The Key-in dialog box will appear with your MDLTaskId and available key-in commands displayed. you can double click a key-in command to test. If you are debugging and have breakpoints set, you can debug those breakpoints in Visual Studio.

Main Program

Program.cs or Program.vb are the main program files containing the main class for the application addins, respectively. MicroStation looks for these classes that is derived from the Bentley.MstnPlatformNET.Addin class. The first line of code within the classes will define your MDLTaskId. The default application addin MDLTaskId is your project name. Also note that your application default namespace is your project name.

CS
[BM.AddIn(MdlTaskID = "MDLTaskId"]
VB .Net
<BM.AddIn(MdlTaskId:="MDLTaskId")>

Unlike MicroStation V8i, if there is an associated keyin command table, the KeyinTree will not be defined in the above code. Instead it will be defined in the project settings .csproj or .vbproj.

<Choose>
<When Condition="Exists('Commands.xml')">
<ItemGroup>
<EmbeddedResource Include="Commands.xml">
<SubType>Designer</SubType>
<LogicalName>CommandTable.xml</LogicalName>
</EmbeddedResource>
</ItemGroup>
</When>
<Choose>

The Commands.xml file is create when a project is created or an item is added with the key-in command table. Keep the naming "Commands.xml" for this purpose.

Please see Program.cs or Program.vb for additonal internal documentation.

Visual Studio Tools for MicroStation Options

From the Visual Studio menu, select Tools>Options>Bentley. Depending on your available installed and licensed Bentley products you can view and change Bentley options. The available option are MicroStation CONNECT Edition, and MicroStation CONNECT Edition Bentley Build (mdl/C/CPP). Please view the available options. You can view your Visual Tools for MicroStation license in the MicroStation CONNECT Edition option page by selecting License Manager.

The Visual Studio Tools for MicroStation projects require options to be set. The options determine the default project settings and properties. Right click the Project in solution explorer and select Properties.

  • Build - Output Path is the MDLAPPS Build Path as defined in the MicroStation CONNECT Edition option page.
  • Build Events - Post-build event is the required post-build event to validate the application and uses the MicroStation Application path and Bentley product as defined in the MicroStation CONNECT Edition option page.
  • Debug - Start external program: is the MicroStation external application path as defined in the option pages.
  • Debug - Command line arguements: is the default debugging arguments as defined in the option pages.
  • Reference Paths - is the default reference paths as defined by the MicroStation application path in the option pages.

Adding Additional References

You can add additional reference paths, if needed, in the project properties page and select Reference Paths page. Click the Select Reference Path button and select the folder you wish to include and click Select Folder. To add references, right click References in solution explorer and select Add References. in the Reference Manager dialog box select Extensions under Assemblies. For those Bentley reference paths set, their Bentley assemblies will be located in Reference Manager>Assemblies>Extensions. For those assemblies not in in a reference path, you need to select Browse in the Reference Manager and browse to and select the appropriate assembly.

Namespace Alias

The application addins generally uses Bentley namespace aliases in the using directives (CSharp) or Imports Statement (VB .Net) to make it easier to qualify an identifier to a Bentley namespace or type. The right side of a using alias directive is always the fully-qualified Bentley type regardless of the using directives that come before it. For example.
C#
instead of
using Bentley.MstnPlatformNET;
We use
using BM = Bentley.MstnPlatformNET;
VB .Net
instead of
Imports Bentley.MstnPlatformNET
We use
Imports BM = Bentley.MstnPlatformNET

Code Snippets

There are CSharp and VB .Net snippets available. The snippets are developed with respect to the default application addin namespace alias'. If you change the application addin namespace alias', the snippets may need modification to reflect the new namespace alias'. In the Visual Studio Code Editor, put the cursor where you want to insert the code snippet. Launch the Insert Code Snippet menu in one of three ways:

  1. Press CTRL+K,CTRL+X.
  2. On the Edit menu, point to IntelliSense, and then click Insert Snippet.
  3. Right-click the mouse and then select the Insert Snippet command on the shortcut menu.

Select the code snippet from the code snippet inserter and then press TAB or ENTER. Alternatively, you can type the name of the code snippet, and then press TAB or ENTER. At the code snippet inserter, select Bentley CONNECT>MicroStation for MicroSation snippets. Select Tools>Code Snippet Manager... and from the Language drop down box select CSharp and review the currently available Bentley CONNECT snippets. Equivalent snippets are available for VB .Net. If you have issues with the code snippets please click here to download MicroStation CONNECT Edition Code Snippets to manually add to the Visual Studio Code Snippet Manager.

New Items

  • MicroStation Class
    MicroStation empty class definition.
  • MicroStation Key-In Command Table
    Adds key-in command table functionality to an existing project that does not already contain key-in command table functionality. Adds xml command table Commands.xml and KeyinCommands.cs or KeyinCommands.vb, regardless of the name in the Add New Item dialog box. Only one key-in command table is allowed per project. With the addition of the Commands.xml file, the .csproj or .vbproj project settings embeds the file as a MicroStation command table as previously noted.
  • MicroStation Placement Tool
    Adds PlacementTool.cs or PlacementTool.vb using the DgnPrimitiveTool base class to implement a primitive command. Placement tools that don't need to locate or modify elements are good candidates for a DgnPrimitiveTool. Inherited abstract members DgnTool.OnDataButton(DgnButtonEvent), DgnPrimitiveTool.OnRestartTool(), and DgnTool.OnResetButton(DgnButtonEvent) must be implemented.
  • MicroStation Placement Tool Settings
    Adds PlacementTool.cs or PlacementTool.vb with an attached form tool setting to implement the DgnPrimitiveTool class.

    The Bentley objects for hooking the tool settings form with MicroStation is in the region 'Initialize MicroStation Tool Setting Form' in the CSharp or Visual Basic form designer, ...ToolSetting.Designer.cs or ...ToolSetting.Designer.vb. The default behavior is a single-instance form. You can modify the MicroStation Form behavior in the ShowForm method.
  • MicroStation Placement Tool Settings Wpf
    Adds PlacementTool.cs or PlacementTool.vb with an attached Window Presentation Form tool setting to implement the DgnPrimitiveTool class.

    The Bentley objects for hooking the tool settings WPF with MicroStation is in the region 'Bentley DockableWindow' in the CSharp or Visual Basic code behind file, ...ToolSetting.xaml.cs or ...ToolSetting.xaml.vb. The default behavior is a single-instance Wpf. You can modify the MicroStation Wpf behavior in the ShowWindow method.
  • MicroStation Selection Tool
    Adds SelectionTool.cs or SelectionTool.vb using the DgnElementSetTool base class to implement a modify command. Inherited abstract members DgnElementSetTool.OnElementModify(Element), and DgnPrimitiveTool.OnRestartTool() must be implemented.
  • MicroStation Selection Tool Settings
    Adds SelectionTool.cs or SelectionTool.vb with an attached form tool setting to implement the IPrimitiveCommandEvents interface.

    The Bentley objects for hooking the tool settings form with MicroStation is in the region 'Initialize MicroStation Tool Setting Form' in the CSharp or Visual Basic form designer, ...ToolSetting.Designer.cs or ...ToolSetting.Designer.vb. The default behavior is a single-instance form. You can modify the MicroStation Form behavior in the ShowForm method.
  • MicroStation Selection Tool Settings Wpf
    Adds SelectionTool.cs or SelectionTool.vb with an attached Windows Presentation Form tool setting to implement the IPrimitiveCommandEvents interface.

    The Bentley objects for hooking the tool settings WPF with MicroStation is in the region 'Bentley DockableWindow' in the CSharp or Visual Basic code behind file, ...ToolSetting.xaml.cs or ...ToolSetting.xaml.vb. The default behavior is a single-instance Wpf. You can modify the MicroStation Wpf behavior in the ShowWindow method.
  • MicroStation UserControl (Wpf)
    Windows Presentation Foundation (Wpf) graphical user interface for developing modern application visual user interfaces. Adds an UserControl.xaml and an associated UserControl.xaml.cs or UserControl.xaml.vb file. In the UserControl.xaml.cs or UserControl.xaml.vb file is a UserControl class derived from the Windows UserControl class. In the ShowWindow method, the private field currentControl is set to Bentley.MstnPlatformNET.WPF.DockableWindow().

    The Bentley objects for hooking the WPF with MicroStation is in the region 'Bentley DockableWindow' in the CSharp or Visual Basic code behind file, UserControl.xaml.cs or UserControl.xaml.vb. The default behavior is a single-instance Wpf. You can modify the MicroStation Wpf behavior in the ShowWindow method.
  • MicroStation Windows Form
    Windows Forms (WinForms) is the classic graphical (GUI) class library included as a part of Microsoft .NET Framework, providing a platform to write rich client applications. Adds a Form.cs or Form.vb and an associated Form.Designer.cs or Form.Designer.vb. The Form.cs or Form.vb Form classes derived from the Bentley.MstnPlatformNET.WinForms.Adapter class which sets the default criteria for the MicroStation form. The Bentley.MstnPlatformNET.WinForms.Adapter is not compatible with the Visual Studio Windows Form designer. To make changes to the Windows Form using the designer right click the Form.cs or Form.vb and select View Code. In the code editor, comment out BMW.Adapter and uncomment //Form. Double click the the Form.cs or Form.vb to enter the designer. When the design is complete, close the designer, In the code editor, comment out Form and uncomment //BMW.Adapter. Form is the name from the Add New Items dialog box.

    The Bentley objects for hooking the form with MicroStation is in the region 'Intialize MicroStation Form' in the CSharp or Visual Basic form designer, Form.Designer.cs or Form.Designer.vb. The default behavior is a single-instance form. You can modify the MicroStation Form behavior in the ShowForm method.
